REQ-BRW-002: 内存渲染

基于 servo 内置 webrender 渲染管线，支持 SoftwareRenderingContext（无头场景）和 servo 原有渲染后端（尊重 servo 原有设计）。通过 take_screenshot() 获取 RgbaImage 后编码输出。

验收标准

  • HTML → CSS 布局 (Stylo) → Display List → webrender 光栅化 → RGBA 像素缓冲管线完整
  • 支持 Flexbox 布局（servo Stylo + LayoutThread 内置）
  • 支持 Grid 布局（servo Stylo + LayoutThread 内置）
  • 支持文字渲染（servo webrender 内置）
  • 输出 PNG 格式截图（RgbaImage → image crate PNG 编码）
  • 输出 JPEG 格式截图（RgbaImage → image crate JPEG 编码）
  • 输出 WebP 格式截图（RgbaImage → image crate WebP 编码）
  • 无头场景支持 SoftwareRenderingContext（纯软件渲染，无需窗口系统）
  • 单次截图延迟 ≤ 500ms (视口 1920×1080)

servo 内置完整渲染管线（Stylo CSS + LayoutThread + webrender 光栅化），尊重 servo 原有设计。 bao_browser 复用 servo 渲染能力：创建 RenderingContext → WebView::paint() → WebView::take_screenshot() → image crate 编码输出。 无头场景使用 SoftwareRenderingContext，有 GPU 时使用 servo 原有 GL 后端。

REQ-BRW-003: SpiderMonkey JSContext 融合

servo 拥有唯一 JSContext，Bao 的 Node.js/Bun host functions 直接注册到 servo 的 JSContext 中。选择 SpiderMonkey 的目的就是与 servo 共享同一引擎，实现真正融合。

架构原则

Bao 是统一库，servo 和 Node.js API 始终共存于同一 JSContext。 servo 的 ScriptThread 创建 JSContext，Bao 的 host functions （console/require/fs/path/crypto/http 等）注册为该 JSContext 的全局对象。 同一个 JSContext 内同时存在 Web API 和 Node.js API，无任何转换层。

验收标准

关联设计

参见 02-SYSTEM.html §3 (Servo 组件架构) · 05-IMPLEMENTATION.html §3 (Phase 2 任务)

关联实体: BaoRuntime

关联测试:

整个 Bao 只有一个 JSContext（servo ScriptThread 创建）
Bao 的 console/require/fs/path 等 Node.js API 在 Node Realm 的 global 上可用
evaluate_js() 可信脚本通过 EnterRealm(Node) 切换到 Node Realm 执行，拥有 Node.js + DOM 全权限
evaluate_js_web() 页面脚本在 Page Realm（Window）执行，只有 Web API，物理上无法访问 Node API
SpiderMonkey Compartment 隔离保证 Page Realm 无法发现 Node Realm 的任何对象
GC 由 servo spin_event_loop() 统一管理，两个 Realm 共享同一 GC
无 JSEngine 重复初始化，Node Realm 通过 JS_NewGlobalObject 在 servo JSContext 上创建

  • close_page(id) 显式关闭 Page，释放 WebView + Realm
  • Page.drop 后 WebView 自动触发 servo CloseWebView 清理
  • 所有 Page 共享同一 Servo 实例和 JSContext

REQ-LIB-002: PagePool 资源管理

实现分层资源管理策略，防止爬虫等长期运行场景下的内存泄漏。

验收标准

  • 活跃 Page（引用计数 > 0）无数量上限
  • 空闲 Page（引用计数 = 0）进入 idle_ttl 倒计时（默认 60s）

  • 空闲回收集成到事件循环 tick 中（每次 spin_event_loop 检查）
  • 累计统计：active / total_created / total_destroyed 可查询
