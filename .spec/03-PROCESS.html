§1 JS 执行管线

数据流图: JS 文件执行

用户代码 (index.ts)
  │
  ├─ [bun_transpiler] TS → JS 转译 (100% 复用 Bun)
  │     输入: .ts/.tsx/.jsx 源码
  │     输出: JS 字节码字符串
  │
  ├─ [bun_resolver] 模块解析 (100% 复用 Bun)
  │     输入: import 路径
  │     输出: 绝对文件路径
  │
  ├─ [bao_engine] SpiderMonkey 编译 + 执行
  │     输入: JS 源码字符串
  │     处理: SM frontend parser → Bytecode → Interpreter/JIT
  │     输出: JS Value (result)
  │
  ├─ [bao_engine] JobQueue 桥接
  │     输入: SM Promise microtask
  │     处理: 注入 Bun event loop tick
  │     输出: 回调执行
  │
  └─ [bun event loop] I/O 处理 (100% 复用 Bun)
        输入: epoll/kqueue 事件
        输出: 回调触发 → SM JobQueue

§5 Event Loop 桥接流程

SpiderMonkey JobQueue ↔ Bun uSockets 桥接

┌─ SpiderMonkey Side ──────────────────────────────┐
│                                                    │
│  JS Promise.then()                                 │
│    → SM 内部队列                                    │
│    → JS::JobQueue::enqueue() ← Bao 实现             │
│                                                    │
└─────────────┬──────────────────────────────────────┘
              │ 桥接点: Bao 实现的 JobQueue trait
              │
┌─────────────┴──────────────────────────────────────┐
│                                                     │
│  Bun Event Loop (uSockets)                          │
│    → epoll/kqueue 事件                              │
│    → Timer 到期                                     │
│    → I/O 就绪                                       │
│    → 每轮 tick 结尾: drain SM JobQueue               │
│    → 执行所有 pending microtasks                     │
│                                                     │
└─────────────────────────────────────────────────────┘

关键约束:
- SM JobQueue 必须在 Bun event loop tick 中被 drain
- 宏任务 (setTimeout) 由 Bun Timer 管理
- 微任务 (Promise.then) 由 SM JobQueue 管理
- 两者共享同一线程 (Main Thread), 无需同步

§6 Module Loader 桥接流程

SpiderMonkey Module Loader ↔ Bun Resolver 桥接

┌─ SpiderMonkey Side ──────────────────────────────────┐
│                                                       │
│  import { foo } from './bar.ts'                       │
│    → SM ESM hook: resolve(specifier, referrer)        │
│    → Bao 实现: 委托给 Bun resolver                    │
│    → SM ESM hook: load(url)                           │
│    → Bao 实现: 委托给 Bun transpiler + FS read        │
│                                                       │
└──────────────┬────────────────────────────────────────┘
               │ 桥接点: Bao 实现的 SM hook callbacks
               │
┌──────────────┴────────────────────────────────────────┐
│                                                        │
│  Bun Resolver (100% 复用)                              │
│    → 路径解析 (node_modules, tsconfig paths, bunfig)  │
│    → 缓存                                              │
│    → 返回绝对路径                                       │
│                                                        │
│  Bun Transpiler (100% 复用)                            │
│    → TS → JS                                           │
│    → JSX → JS                                          │
│    → 返回 JS 源码字符串                                  │
│                                                        │
└────────────────────────────────────────────────────────┘

关键约束:
- SM hook 必须在 JSContext 初始化时注册
- Bun resolver 是同步的, SM hook 必须也是同步的
- 文件读取可以异步, 但 module source 返回必须同步
- 缓存机制: resolved URL → transpiled source 映射表

§8 线程模型与数据流

线程间数据流

Main Thread (事件循环 + JSContext + DOM)
  │
  ├─ 单线程访问 SpiderMonkey JSContext
  ├─ 单线程访问 libservo DOM (script 组件)
  ├─ CDP 命令通过 crossbeam channel 接收
  │
  ├─→ CDP Thread (WebSocket 服务)
  │     ├─ 接受 CDP 客户端连接
  │     ├─ JSON-RPC 解析/编码
  │     └─→ Main Thread: 命令通过 channel 发送
  │
  ├─→ Render Thread (servo webrender)
  │     ├─ 接收 Display List
  │     ├─ webrender 光栅化
  │     └─→ Main Thread: 返回像素缓冲
  │
  ├─→ Network Thread Pool (HTTP 请求)
  │     ├─ Bun HTTP 栈处理
  │     └─→ Main Thread: 回调通过 event loop
  │
  └─→ Worker Threads (Web Workers)
        ├─ 每个线程独立 JSContext
        └─→ Main Thread: postMessage 通过 channel

线程安全约束:
- JSContext: 仅 Main Thread 访问
- DOM: 仅 Main Thread 访问 (libservo script 组件单线程)
- 渲染: Display List 生成后传递给 Render Thread
- 网络: Bun HTTP 栈内部处理线程安全
- CDP: 命令路由到 Main Thread, 响应路由回 CDP Thread

Page 创建流程时序图

调用方: runtime.createPage(config)
  │
  ├─ PagePool: 检查 max_total 硬上限
  │   └─ 超出 → 返回 Err(PageLimitExceeded)
  │
  ├─ 创建 SoftwareRenderingContext (独立渲染缓冲)
  │   └─ PhysicalSize::new(config.viewport_width, config.viewport_height)
  │
  ├─ 创建 BaoWebViewDelegate (独立状态回调)
  │   └─ Rc::new(BaoWebViewDelegate::new(state, viewport))
  │
  ├─ WebViewBuilder::new(&servo, rendering_context)
  │   ├─ .delegate(delegate)
  │   ├─ .url(url) (如果配置了初始 URL)
  │   └─ .build()
  │
  ├─ 构造 PageHandle
  │   ├─ id: WebViewId (自动生成)
  │   ├─ webview: WebView
  │   ├─ rendering_context
  │   ├─ delegate
  │   ├─ state: Created
  │   ├─ permission: config.permission
  │   ├─ stealth_profile: config.stealth_profile
  │   ├─ last_active_at: now()
  │   └─ created_at: now()
  │
  ├─ PagePool: 插入 active_pages HashMap
  │
  ├─ 如果 config.stealth_profile 存在:
  │   └─ StealthEngine::inject_navigator_js() → webview.evaluate_javascript()
  │
  └─ 返回 Ok(Rc<PageHandle>)

CDP 路由分发流程时序图

调用方: cdp.send('Domain.method', params)
  │
  ├─ CdpRouter: 解析 method → Domain + MethodName
  │
  ├─ 检查后端类型
  │   │
  │   ├─ [Internal] CdpBackendInternal
  │   │   ├─ 查找 Domain Handler
  │   │   ├─ 定位目标 PageHandle (通过 session → target_id)
  │   │   ├─ 直接调用 bao_browser API
  │   │   │   ├─ Page.navigate → webview.load(url)
  │   │   │   ├─ Page.captureScreenshot → webview.paint() + take_screenshot()
  │   │   │   ├─ Runtime.evaluate → webview.evaluate_javascript()
  │   │   │   ├─ DOM.querySelector → webview.evaluate_javascript(dom_query_js)
  │   │   │   └─ Network.* → Bun HTTP 栈 hook
  │   │   └─ 返回结果（零序列化，直接 Rust 值）
  │   │
  │   └─ [External] CdpBackendExternal
  │       ├─ 序列化为 JSON-RPC 2.0
  │       ├─ 通过 WebSocket 发送到外部端点
  │       ├─ 等待响应（超时 10s）
  │       └─ 反序列化并返回
  │
  └─ 返回 CdpResponse / CdpError

PagePool 空闲回收流程时序图

事件循环每轮 tick:
  │
  ├─ PagePool::check_idle_pages()
  │   │
  │   ├─ 遍历 idle_pages HashMap
  │   │   │
  │   │   └─ 对每个 (page_id, (page, idle_since)):
  │   │       ├─ now - idle_since > idle_ttl?
  │   │       │   ├─ 否 → 跳过
  │   │       │   └─ 是 → 执行回收:
  │   │       │       ├─ page.state = Closing
  │   │       │       ├─ WebView::drop (触发 servo CloseWebView)
  │   │       │       ├─ RenderingContext 释放
  │   │       │       ├─ SM Realm GC 回收 (JSContext 统一 GC)
  │   │       │       ├─ PixelBuffer 释放
  │   │       │       ├─ 从 idle_pages 移除
  │   │       │       ├─ stats.total_destroyed += 1
  │   │       │       └─ page.state = Closed
  │   │       │
  │   │       └─ 继续检查下一个
  │   │
  │   └─ 返回 (回收的页面数)
  │
  └─ 继续事件循环其他 tick 任务

§10 异常流程

异常处理流程

JS 执行异常:
  SM 抛出 JS Exception → Bao bridge 检测
    → 未捕获: 输出 stack trace 到 stderr
    → CDP Runtime: 发送 Runtime.exceptionThrown 事件
    → 进程退出码 1

DOM 解析异常:
  html5ever 容错解析 → 生成错误恢复 DOM
    → 不中断渲染流程
    → CDP Console: 发送 Console.error 事件

网络请求异常:
  Bun HTTP 栈 → 超时/连接拒绝/DNS 失败
    → 返回 NetworkError
    → CDP Network: 发送 Network.loadingFailed 事件
    → JS fetch: 返回 rejected Promise

渲染异常:
  servo webrender 渲染失败
    → 返回空像素缓冲 (全白/全透明)
    → CDP Page.captureScreenshot: 返回空白截图
    → 日志记录错误

CDP 协议异常:
  未知 domain/method
    → 返回 JSON-RPC Error (-32601 Method not found)
  参数错误
    → 返回 JSON-RPC Error (-32602 Invalid params)
  内部错误
    → 返回 JSON-RPC Error (-32603 Internal error)

SpiderMonkey GC OOM:
  SM GC 报告内存不足
    → 触发紧急 GC
    → 仍然不足: 抛出 OOM Error
    → 进程退出码 134 (SIGABRT)
